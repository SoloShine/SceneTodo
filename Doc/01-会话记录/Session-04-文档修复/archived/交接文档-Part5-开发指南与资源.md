## 📏 开发规范与最佳实践

### 核心规范文档

SceneTodo 项目遵循一套完整的开发规范，确保代码质量和可维护性。

**📖 完整规范文档**: [SceneTodo开发规范与最佳实践](../05-开发文档/SceneTodo开发规范与最佳实践.md)

### 重点规范速览

#### 1. 代码组织规范
- 按功能模块组织代码
- 单一职责原则
- 高内聚低耦合

#### 2. 分部类使用规范 ⭐ 重要

**何时使用**:
- 单个类超过 500 行
- 类有多个清晰的功能模块
- 自动生成的代码（如 Designer.cs）

**命名格式**: `<ClassName>.<Feature>.cs`

**示例**:
```
ViewModels/
├── MainWindowViewModel.Core.cs              # 核心定义
├── MainWindowViewModel.Navigation.cs        # 导航功能
├── MainWindowViewModel.TodoManagement.cs    # 待办管理
└── ...
```

**详细文档**: [MainWindowViewModel分部类重构文档](../05-开发文档/MainWindowViewModel分部类重构文档.md)

#### 3. 命名规范
- **类名**: PascalCase (`TodoItemModel`)
- **方法**: PascalCase (`SaveData`)
- **变量**: camelCase (`userName`)
- **私有字段**: 下划线前缀 (`_service`)
- **常量**: PascalCase 或 UPPER_CASE

#### 4. MVVM 架构规范
- View 只负责显示
- ViewModel 处理业务逻辑
- Model 管理数据
- 使用数据绑定和命令

#### 5. 异步编程规范
- 异步方法以 `Async` 结尾
- 使用 `await` 而不是 `.Wait()`
- 在库代码中使用 `ConfigureAwait(false)`

#### 6. Git 提交规范
- 格式: `<type>(<scope>): <subject>`
- Type: feat, fix, docs, style, refactor, perf, test, chore
- 一次提交只做一件事

### 代码质量检查清单

提交代码前请检查：

- [ ] ✅ 符合命名规范
- [ ] ✅ 有适当的注释
- [ ] ✅ 没有编译警告
- [ ] ✅ 单个类不超过 500 行（考虑使用分部类）
- [ ] ✅ 方法不超过 100 行
- [ ] ✅ 遵循 MVVM 模式
- [ ] ✅ 异步方法正确使用
- [ ] ✅ 提交消息符合规范

### 常见问题

**Q: 什么时候应该使用分部类？**  
A: 当单个类超过 500 行，或者有多个清晰的功能模块时。

**Q: 如何组织分部类文件？**  
A: 创建 Core 文件存放字段、属性、命令，创建功能文件存放方法实现。

**Q: 分部类文件如何命名？**  
A: 格式为 `ClassName.Feature.cs`，如 `MainWindowViewModel.TodoManagement.cs`。
